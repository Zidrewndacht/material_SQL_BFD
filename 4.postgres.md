## 0: Contexto Arquitetural

O **PostgreSQL** é um DBMS relacional de código aberto, padrão da indústria, que opera sob a arquitetura **Cliente-Servidor**. 

*(Respondendo à dúvida comum: "É o mais utilizado em aplicações web modernas?" — **Sim.** É considerado o SGBD relacional de código aberto mais avançado do mundo e é a espinha dorsal de backends de alta escala em empresas como Instagram, Spotify, Apple, Reddit e Netflix. Quando a arquitetura exige um banco relacional robusto para a web, o PostgreSQL é a escolha padrão do mercado.)*

---

## 1: O que Permanece Igual

SQL é uma linguagem padronizada (ANSI/ISO). A transição do SQLite para o PostgreSQL preserva a maior parte do conhecimento adquirido:

| Conceito | SQLite | PostgreSQL |
| :--- | :--- | :--- |
| `CREATE TABLE`, `INSERT`, `SELECT`, `UPDATE`, `DELETE` | ✅ | ✅ (sintaxe idêntica) |
| `INNER JOIN`, `LEFT JOIN` | ✅ | ✅ (sintaxe idêntica) |
| `PRIMARY KEY`, `FOREIGN KEY`, `NOT NULL`, `UNIQUE` | ✅ | ✅ (sintaxe idêntica) |
| `WHERE`, `ORDER BY`, `GROUP BY`, `HAVING` | ✅ | ✅ (sintaxe idêntica) |
| Queries parametrizadas (proteção contra SQL Injection) | ✅ | ✅ (sintaxe do placeholder muda — ver 4) |
| Transações (`COMMIT`, `ROLLBACK`) | ✅ | ✅ (no PostgreSQL são explícitas e obrigatórias) |
| Acesso por nome de coluna | ✅ (via `sqlite3.Row`) | ✅ (via `RealDictCursor` do `psycopg2`) |

## 2: O que Muda — Arquitetura Cliente-Servidor

### 2.1 Diferença Arquitetural Fundamental

| Aspecto | SQLite (Embarcado) | PostgreSQL (Cliente-Servidor) |
| :--- | :--- | :--- |
| **Processo** | Biblioteca embutida no seu script Python. | Serviço independente rodando como processo no SO (ou em um servidor remoto). |
| **Comunicação** | Chamadas de função diretas na memória. | Protocolo de rede TCP/IP (porta padrão: `5432`). |
| **Armazenamento** | Um único arquivo `.db` no sistema de arquivos. | Diretório de dados gerenciado exclusivamente pelo processo do servidor. |
| **Autenticação** | Nenhuma (quem tem acesso ao arquivo, tem acesso ao banco). | Usuário e senha obrigatórios por conexão (Roles). |
| **Concorrência** | Um único escritor por vez (lock de arquivo). | Múltiplos leitores e escritores simultâneos via **MVCC** (*Multi-Version Concurrency Control*). |
| **Instalação** | Zero (nativo no Python). | Requer instalação do servidor + driver Python. |

#### 2.1.1 O Conceito de MVCC (Multi-Version Concurrency Control)
Na tabela acima, mencionamos que o PostgreSQL utiliza **MVCC** para lidar com concorrência, enquanto o SQLite utiliza *locks* (travamentos) de arquivo. É fundamental compreender a diferença arquitetural entre essas duas abordagens:

*   **A Abordagem do SQLite (Locking):** Quando um processo precisa escrever no banco, o SQLite frequentemente precisa travar o arquivo inteiro (ou partes dele). Se o Usuário A está escrevendo, o Usuário B que tentar ler ou escrever ao mesmo tempo receberá o erro `database is locked` e terá que esperar ou falhar.
*   **A Abordagem do PostgreSQL (MVCC):** O PostgreSQL nunca sobrescreve dados diretamente no disco. Quando o Usuário B atualiza uma linha, o banco cria uma **nova versão** dessa linha no disco, mantendo a versão antiga intacta temporariamente. 
    *   Se o Usuário A iniciou uma leitura *antes* da atualização do Usuário B, o PostgreSQL garante que o Usuário A continue lendo a **versão antiga** (o *snapshot* do momento em que a leitura começou).
    *   **A Regra de Ouro do MVCC:** *"Leitores não bloqueiam escritores, e escritores não bloqueiam leitores."* Isso permite que aplicações web com milhares de usuários simultâneos leiam e escrevam dados ao mesmo tempo sem que o sistema trave ou retorne erros de concorrência constantes.


### 2.2 Implicações Práticas
*   O banco de dados pode rodar em uma máquina diferente da aplicação (ex: banco na AWS, aplicação no seu notebook).
*   É necessário gerenciar **usuários** e **permissões** no banco.
*   A conexão é um recurso de rede: pode falhar, ter timeout e precisa ser gerenciada.
*   Não existe mais um "arquivo" para copiar, enviar por e-mail ou abrir no DB Browser diretamente.

## 3: Instalação e Configuração

### 3.1 Instalando o Servidor PostgreSQL (Windows)

1. Acesse o site oficial: [postgresql.org/download/windows](https://www.postgresql.org/download/windows/)
2. Baixe o instalador fornecido pela EnterpriseDB (EDB).
3. Durante a instalação:
   *   Anote a **porta** (padrão: `5432`).
   *   Defina a **senha do usuário `postgres`** (superusuário).
   *   O instalador configurará o PostgreSQL como um **serviço do Windows** (inicia automaticamente com o sistema).
4. O instalador inclui o **pgAdmin 4** (ferramenta visual de administração — ver 8).

> **Nota sobre Docker:** No mercado de trabalho, é muito comum que desenvolvedores não instalem o Postgres diretamente no Windows, mas sim o executem via **Docker** (`docker run --name postgres -e POSTGRES_PASSWORD=123 -p 5432:5432 -d postgres`). Isso mantém o sistema operacional limpo e facilita a configuração de ambientes.

### 3.2 Criando o Banco de Dados e o Usuário

Após a instalação, abra o **pgAdmin 4** ou o terminal `psql` (incluído na instalação) e execute:

```sql
-- Cria um usuário dedicado à aplicação (nunca use 'postgres' em produção)
CREATE USER app_user WITH PASSWORD 'senha_segura_123';

-- Cria o banco de dados
CREATE DATABASE loja_db OWNER app_user;

-- Concede privilégios
GRANT ALL PRIVILEGES ON DATABASE loja_db TO app_user;
```

### 3.3 Instalando o Driver Python (`psycopg2`)

O `psycopg2` é o driver (adaptador) mais utilizado para comunicação entre Python e PostgreSQL. Diferente do `sqlite3`, ele é um pacote de terceiros e precisa ser instalado no ambiente virtual.

```powershell
# No terminal, com o ambiente virtual (.venv) ativo:
pip install psycopg2-binary
```

> **Nota técnica:** O pacote `psycopg2-binary` inclui as dependências C pré-compiladas, facilitando a instalação em Windows e macOS. Em ambientes de produção Linux (Docker/Servidores), a boa prática é compilar o `psycopg2` a partir do fonte (`pip install psycopg2`) para garantir performance e compatibilidade com a biblioteca `libpq` do sistema.

## 4: Diferenças Práticas de Sintaxe

### 4.1 String de Conexão
No SQLite, a conexão recebe um caminho de arquivo. No PostgreSQL, recebe os parâmetros de rede e autenticação.

```python
import psycopg2

conexao = psycopg2.connect(
    host="localhost",       # Endereço do servidor (pode ser um IP remoto)
    port="5432",            # Porta do serviço PostgreSQL
    database="loja_db",     # Nome do banco de dados
    user="app_user",        # Usuário autenticado
    password="senha_segura_123"  # Senha do usuário
)
```

### 4.2 Placeholder de Parametrização
O `sqlite3` usa `?`. O `psycopg2` usa `%s` (independente do tipo de dado).

```python
# PostgreSQL:
cursor.execute("INSERT INTO clientes (nome) VALUES (%s)", ("Ana",))
```
*Atenção: `%s` no `psycopg2` não é formatação de string Python. É o marcador de posição do driver. Nunca use f-strings para montar queries.*

### 4.3 AUTOINCREMENT vs. SERIAL / IDENTITY
O SQLite usa `AUTOINCREMENT`. O PostgreSQL tradicionalmente usa o tipo `SERIAL` (que cria uma sequência por baixo dos panos) ou a sintaxe moderna do padrão SQL `GENERATED ALWAYS AS IDENTITY`.

```sql
CREATE TABLE clientes (
    id SERIAL PRIMARY KEY,
    nome TEXT NOT NULL
);
```

### 4.4 Transações e o Gerenciador de Contexto (`with`)

No PostgreSQL, **toda operação ocorre dentro de uma transação**. O driver `psycopg2` inicia uma transação implicitamente no momento em que você executa o primeiro comando (como um `INSERT` ou `UPDATE`). A partir desse momento, o banco de dados mantém as alterações em um estado pendente até que você decida o destino delas.

#### A "Pegadinha" do `with` no psycopg2
Nas aulas anteriores, usamos o `with` para manipulação de arquivos (`with open(...)`) e no `sqlite3`. É comum assumir que o `with` serve para "fechar" o recurso ao final do bloco. **No `psycopg2`, isso não é verdade.**

O gerenciador de contexto (`with conexao:`) no `psycopg2` serve **exclusivamente para delimitar o escopo da transação** (garantindo o *commit* ou o *rollback*). Ele **não encerra a conexão de rede** com o servidor.

**O que o Python executa por baixo dos panos:**
```python
# O que você escreve:
with conexao:
    with conexao.cursor() as cursor:
        cursor.execute("INSERT INTO clientes (nome) VALUES (%s)", ("Ana",))

# O que o psycopg2 executa logicamente:
try:
    # O cursor executa a query. A transação já foi iniciada implicitamente.
    cursor.execute("INSERT INTO clientes (nome) VALUES (%s)", ("Ana",))
except Exception:
    conexao.rollback()  # Se der erro, desfaz TUDO o que ocorreu no bloco 'with'
    raise
else:
    conexao.commit()    # Se passar sem erros, salva as alterações no servidor
```

#### O Ciclo de Vida Completo da Conexão
Como o `with` não fecha a conexão, você ainda precisa gerenciá-la manualmente para evitar o esgotamento do *pool* de conexões do servidor (vazamento de recursos). A estrutura robusta e recomendada para operações no `psycopg2` é:

```python
conexao = psycopg2.connect(**DB_CONFIG)
try:
    # O 'with' aqui gerencia APENAS a transação (commit/rollback)
    with conexao:
        with conexao.cursor() as cursor:
            cursor.execute("UPDATE produtos SET estoque = estoque - 1 WHERE id = %s", (1,))
            # Se uma exceção ocorrer aqui, o rollback é automático.
            
except psycopg2.Error as e:
    print(f"Erro de banco de dados: {e}")
finally:
    # O 'finally' garante que a conexão de rede seja encerrada, 
    # independentemente de a transação ter tido sucesso ou falhado.
    conexao.close()
```

*Nota técnica:* O bloco interno `with conexao.cursor() as cursor:` é um gerenciador de contexto diferente, fornecido pelo `psycopg2`, que serve para **fechar o cursor** (liberar a memória do lado do servidor associada àquela consulta específica) ao final do bloco, mas também não fecha a conexão principal.

### 4.5 PRAGMA foreign_keys
No SQLite, a integridade referencial precisava ser ativada com `PRAGMA`. No PostgreSQL, **Foreign Keys são validadas nativamente e sempre estão ativas**. O `PRAGMA` não existe e não é necessário.

### 4.6 `executescript`
O método `executescript` não existe no `psycopg2`. Para executar múltiplos comandos DDL, utiliza-se `cursor.execute()` com uma string contendo múltiplos statements separados por `;`.

## 5: Recursos Exclusivos do PostgreSQL

É aqui que o PostgreSQL justifica sua complexidade de instalação, oferecendo recursos que resolvem problemas que, em outros bancos, exigiriam softwares externos.

### 5.1 Tipos de Dados Avançados
| Tipo | Descrição | Exemplo de Uso |
| :--- | :--- | :--- |
| `VARCHAR(n)` | Texto com limite máximo de caracteres. | `email VARCHAR(255)` |
| `TIMESTAMPTZ` | Data e hora com consciência de fuso horário. | `criado_em TIMESTAMPTZ DEFAULT NOW()` |
| `UUID` | Identificador universal único (128-bit). | `id UUID PRIMARY KEY` |
| `ARRAY` | Coluna que armazena uma lista de valores. | `tags TEXT[]` |
| `BOOLEAN` | Tipo booleano nativo (`true`/`false`). | `ativo BOOLEAN DEFAULT true` |

### 5.2 JSONB (Binary JSON)
Permite armazenar documentos JSON não estruturados e, crucialmente, **criar índices (GIN)** sobre as chaves internas desse JSON. Isso une a flexibilidade do NoSQL (como MongoDB) com a performance e integridade dos DBMS.

```sql
-- Busca indexada dentro de um JSON
SELECT * FROM produtos WHERE metadados @> '{"cor": "preto"}';
```

### 5.3 `RETURNING`
Permite que comandos `INSERT`, `UPDATE` e `DELETE` retornem as linhas afetadas, eliminando a necessidade de uma consulta subsequente para obter o ID gerado (substitui o `cursor.lastrowid` do SQLite).

### 5.4 Full-Text Search e PostGIS
*   **Full-Text Search:** Busca textual nativa com *stemming* (redução de palavras à raiz) e ranqueamento de relevância, eliminando a necessidade de integrar um ElasticSearch para volumes médios.
*   **PostGIS:** Extensão geoespacial que transforma o Postgres no banco de dados geográfico mais poderoso do mundo, permitindo queries como *"Selecione todos os motoboys a menos de 2km de raio da coordenada X"*.

## 6: O que se "Perde" na Migração (Trade-offs)

A migração para um DBMS Cliente-Servidor não é uma evolução linear. Há custos operacionais reais:

| Aspecto | Impacto da Migração |
| :--- | :--- |
| **Complexidade de Setup** | Exige instalação do servidor, criação de usuários, configuração de rede. |
| **Portabilidade** | Não é mais possível "copiar um arquivo e enviar por email". Backups exigem `pg_dump`. |
| **Ambiente de Desenvolvimento** | Cada desenvolvedor da equipe precisa ter o PostgreSQL rodando localmente (ou via Docker). |
| **Dependência Externa** | Se o serviço do PostgreSQL parar, a aplicação para. No SQLite, o banco "está sempre lá". |
| **Gerenciamento de Credenciais** | Senhas de banco nunca devem estar hardcoded no código. Exigem variáveis de ambiente (`os.environ`). |
| **Latência de Rede** | Mesmo em `localhost`, a comunicação via TCP/IP é mais lenta que uma chamada de função na memória. |

## 7: Exemplo Guiado — Catálogo de Vendas em PostgreSQL

Reimplementação do exemplo da Aula 10, adaptado para PostgreSQL com `psycopg2`.

```python
import psycopg2
from psycopg2.extras import RealDictCursor

# 1. Estabelece a conexão com o servidor PostgreSQL.
# Em produção, esses valores viriam de variáveis de ambiente (os.environ).
conexao = psycopg2.connect(
    host="localhost",
    port="5432",
    database="loja_db",
    user="app_user",
    password="senha_segura_123"
)

# 2. RealDictCursor equivale ao sqlite3.Row: permite acesso por nome de coluna.
cursor = conexao.cursor(cursor_factory=RealDictCursor)

# 3. Criação do Schema (DDL).
# 'SERIAL' substitui 'INTEGER PRIMARY KEY AUTOINCREMENT'.
# Foreign Keys são validadas automaticamente (sem PRAGMA).
cursor.execute("""
    CREATE TABLE IF NOT EXISTS clientes (
        id SERIAL PRIMARY KEY,
        nome VARCHAR(100) NOT NULL
    )
""")

cursor.execute("""
    CREATE TABLE IF NOT EXISTS pedidos (
        id SERIAL PRIMARY KEY,
        cliente_id INTEGER NOT NULL,
        produto VARCHAR(200) NOT NULL,
        FOREIGN KEY (cliente_id) REFERENCES clientes(id)
    )
""")
# Confirma a criação das tabelas no banco.
conexao.commit()

# 4. Inserção com RETURNING (captura o ID gerado sem consulta extra).
cursor.execute(
    "INSERT INTO clientes (nome) VALUES (%s) RETURNING id",
    ("Ana Silva",)
)
# fetchone() retorna um dicionário: {'id': 1}
id_ana = cursor.fetchone()["id"]

cursor.execute(
    "INSERT INTO clientes (nome) VALUES (%s) RETURNING id",
    ("Bruno Costa",)
)
id_bruno = cursor.fetchone()["id"]

conexao.commit()

# 5. Inserção de pedidos usando os IDs dinâmicos capturados.
# Note o placeholder %s (padrão do psycopg2, independente do tipo de dado).
cursor.execute(
    "INSERT INTO pedidos (cliente_id, produto) VALUES (%s, %s)",
    (id_ana, "Teclado Mecânico")
)
cursor.execute(
    "INSERT INTO pedidos (cliente_id, produto) VALUES (%s, %s)",
    (id_ana, "Mouse Gamer")
)
conexao.commit()

# 6. Teste de Integridade Referencial (validação nativa do PostgreSQL).
try:
    cursor.execute(
        "INSERT INTO pedidos (cliente_id, produto) VALUES (%s, %s)",
        (9999, "Monitor")
    )
    conexao.commit()
except psycopg2.IntegrityError as e:
    # O PostgreSQL rejeita a operação e o driver levanta IntegrityError.
    conexao.rollback()  # OBRIGATÓRIO: sem rollback, a conexão fica em estado de erro.
    print(f"[Integridade] Bloqueado com sucesso: {e}")

# 7. Consulta com INNER JOIN.
# A sintaxe do SQL é idêntica à usada com SQLite.
cursor.execute("""
    SELECT c.nome, p.produto
    FROM clientes c
    INNER JOIN pedidos p ON c.id = p.cliente_id
    ORDER BY c.nome
""")

print("\n--- INNER JOIN ---")
for row in cursor.fetchall():
    # 'row' é um dicionário: {'nome': 'Ana Silva', 'produto': 'Teclado Mecânico'}
    print(f"{row['nome']} comprou: {row['produto']}")

# 8. Encerramento.
# No psycopg2, é necessário fechar cursor e conexão explicitamente.
cursor.close()
conexao.close()
```

## 8: pgAdmin 4

O **pgAdmin 4** é uma ferramenta gráfica oficial para administração do PostgreSQL, equivalente funcional do DB Browser for SQLite.
*   **Incluso** na instalação padrão do PostgreSQL para Windows.
*   **Acesso:** Iniciar Menu → pgAdmin 4 (abre como aplicação web local no navegador).
*   **Funcionalidades:** Browser de objetos em árvore, Query Tool com autocompletar, Visualizador de Dados e Dashboard de monitoramento.
